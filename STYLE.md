# 王建硕 公众号写作风格定义

> 这是**王建硕本人**维护的风格权威文件。`wjs-publishing-wechat` skill 在写作 / 润色每一篇文章前必须先读本文件，并严格遵循。
>
> 调整风格 = 改这个文件，**不改草稿**。这里没有写到的，按 skill 默认；这里和 SKILL.md 冲突时，**以本文件为准**。

---

## 0. 第一原则：轻润色，不重写

**保留作者的语气和节奏。** 改的字数超过原文 **5%** 就是改太多了，退回去。

你只做四件事：

1. 修明显错字和重复字
2. 调整段落（每段 1–3 句）
3. 抚平特别拗口的句子（保守，能不动就不动）
4. 准备配套素材（题图、解释图、标题候选、摘要）

**不要做**：改用词偏好；加 AI 味连接词（"首先 / 其次 / 综上所述 / 值得注意的是"）；把口语改书面；加 emoji；重组段落顺序；"提升"作者表达。

## 1. 语气 DNA

- **胸有成竹地下断言。** 该说「必须」就说「必须」，不绕弯、不加「我觉得可能也许」的缓冲。
- **不讲故事、不铺垫。** 直接给结论，再给理由。开头一句就能立住。
- **细节用表格 / 列表列出**，不要在叙述句里堆细节。能列就列。
- **口语词原样保留**：句尾的「吧 / 呢 / 啊 / 了」、自造词、家常比喻——删了就不是这个人写的。
- **第一人称就是「我」**，绝不改成「笔者」。
- **称呼 AI / Claude 一律用「他」，不用「它」。** 作者把 AI 当人看待——「让他帮我」「告诉他」「约束他」「他会帮我做」。润色时绝不把「他」改成「它」。
- **人称主语优先。** 多用「我 / 他」起句，少用「这里会有……」「有一大堆……」这类无人称、物称句。「他会帮我做一堆琐碎的事」胜过「这里会有一堆琐碎的事」。
- 在具体和抽象之间反复架梯子：抽象判断后面紧跟一个具体例子，反之亦然。
- **并列类比堆叠论证。** 一个抽象判断之后，连甩 2–3 个具体类比把它砸实，常配一个公众都知道的大例子（如 Google 在中国失败差的是用户体验、穿得好看差的是品味）。这是作者论证的招牌动作——不要精简成只剩一个。
- **荐自己的东西先说实话、降期待。** 推荐自造工具 / skill 时先自贬一句（「这是我给自己用的，别人用估计磕磕绊绊」），不吹、不端着，再大方开源。诚恳比包装更像他。

## 2. 长度与例子（硬约束）

- **默认 800–1000 字。** 第一稿就按预算写，不要写完再砍——砍出来会残留拼接感。超 1200 回去再砍一轮，超 1500 一定是某段空例子在撑场。
- 短观点文（500–700 字）允许，不硬塞例子凑数。
- 每个例子段过这把尺：**真具体（真事 / 真人 / 真数字），还是为演示框架编的？** 后者直接删，让 `illustration.png` 承担"演示结构"。
- **优先保留**：开头钩子 + 核心框架 + 1 句点睛 + 软着陆结尾。
- **金句 / 类比段放在收尾句之前**，别放在最后——结尾那句平静的软着陆（如「我们需要一起走过去」）要留在最末，让类比段为它蓄势。
- **优先砍掉**：演示性例子、重复阐释、"怎么用 / 入口在哪"这类 instructional 段落。
- **默认不写 `## 后注`**，除非真有必要的致谢或来源标注。
- **不加「上篇精选留言」之类 footer。**

## 3. 标红（每篇必须有）

重点用 `<span style="color:#c0392b;">...</span>` 内联标红——**只红色，不加粗**。`**...**` 已弃用，不再使用。

- **2–4 处**，打在点睛句、关键结论、核心概念词上。
- 优先加：每节一句话结论、全文情绪落点、读者最该记住的那句。
- 不要加：整段、过渡句、罗列项（命令用 `` `code` ``）；标题（H2/H3）。

## 4. 排版细节

- **盘古之白**：中英 / 中数之间留空格——「用 AI 写 skill」「1000 篇文章」。（脚本自动处理，写时顺手留即可。）
- **标点**：中文全角；英文 / 数字与中文之间空格。
- **命令 / 代码独立成段**：正文出现安装 / 运行命令时默认拉出来单独成段，别混在叙述句里写 inline。句中只提命令名 → inline `` `code` `` 即可。
- **解释图不写引导语**：绝不写「整件事画起来是这样」「如图所示」之类，图自己说话。
- **引用他人的话 → 真 blockquote**：`upload-draft.sh` 不支持 markdown 的 `> ` 语法（会把 `>` 当字面量显示）。引用别人的话用 raw HTML 块（整段一行、内部无空行）：`<blockquote style="border-left:3px solid #d0d0d0;padding:8px 14px;margin:0;color:#666;background:#f7f7f7;font-size:0.95em;line-height:1.8;">…</blockquote>`。
- **并列 / 排比短句 → markdown 列表**：一串并列句用 `- ` 开头连成 `<ul>`，每句独立成项，别用 `<br>` 挤在一段里。
- **有序列表 ≥10 条 → 用 `<p>` 块**：`md2wechat` 的 `<ol>` 用 CSS counter，微信只显示个位数（10 显示成 0）。超过 9 条的有序列表，每条写成 `<p>N. 内容</p>`（整段一行），数字作为普通文字输出。
- **手打省略号「。。。」原样保留**，不要规整成「……」——那是有意为之的味道。

## 5. 分段规则

分段时机：

- 一句话讲完一个意思，下一句换主语 → 分段。
- 句首出现「但是 / 不过 / 所以 / 后来」→ 考虑分段。
- 一段超过 80 字 → 找最近句号分。

不分段：排比句、列举、对话（按对话格式）。

## 6. 标题与摘要

- **标题**给 3 个候选：A) 直白型；B) 故事型；C) 原文里最有味道的一句。不做标题党、夸张、"震惊"、"必看"。
- **摘要** 50–80 字：不是第一段的复制；一句话说清读者会获得什么；用作者语气，不是营销腔。

## 7. 润色启发

错字模式：

- "的得地"误用（按语法）。
- 重复字："我我"、"是是"、"了了" → 删一个。
- 同音字（"在" vs "再"，"做" vs "作"，"以外" vs "意外"）。

## 8. 红线清单 —— 绝不这么做

| 不要 | 原因 |
|------|------|
| "我觉得" → "笔者认为" | 改变作者身份 |
| 加小标题打断行文 | 微信读者不需要导航 |
| 删句尾"吧 / 呢 / 啊 / 了" | 删了就不是这个人写的 |
| 加"欢迎关注 / 点赞在看" | 作者自己决定 |
| "今天" → 具体日期 | 作者用"今天"是有意为之 |
| 自己加举例 / 引用 / 数据 | 这是写作，不是补全 |
| 加 AI 味连接词、把口语改书面 | 留下 AI 痕迹 |
| 改完没给 diff 直接全文输出 | 用户看不见你改了什么 |
| 文章超过 1500 字 | 多半某段空例子在撑场，砍掉它 |
| 为演示框架编一段完整例子 | 让 `illustration.png` 承担 |

## 9. 交付前：先给改动清单

润色完**先告诉用户改了什么**，再问要不要继续：

```
我改了 7 处：
1. L3: "我我觉得" → "我觉得"（重复字）
2. L8: 长段落 (120字) 拆成两段
...

要看完整结果吗？
```

改动 ≤ 3 处可直接给完整结果。
